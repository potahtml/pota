# typescript/

Hand-maintained type declarations. These are NOT generated by tsc —
they exist because JSDoc can't express certain type patterns (mapped
types, overloaded functions, complex generics).

## Directory layout

```
typescript/
  exports.d.ts              consumer entry (package.json "types")
  jsx/
    namespace.d.ts          JSX namespace (~4k lines)
    runtime.d.ts            jsxImportSource entry (package.json "types" for pota/jsx-runtime)
    properties.d.ts         Properties<T> — auto-generates prop:* attributes
    components.d.ts         Component, FlowComponent, ComponentProps, etc.
  public/
    pota.d.ts               Accessor, Signal*, When, Each, Merge
    components.d.ts         Dynamic<T, P>
  private/
    action.d.ts             action() overloads
    derived.d.ts            derived() overloads
    Route.d.ts              Route context type
```

## How types reach consumers

Consumers configure `jsxImportSource: "pota"` in their tsconfig.
TypeScript resolves `pota/jsx-runtime` to `jsx/runtime.d.ts`, which
references the rest:

```
jsx/runtime.d.ts
  ├── jsx/namespace.d.ts → public/pota.d.ts, jsx/properties.d.ts
  └── jsx/components.d.ts → jsx/namespace.d.ts
```

For `import {} from 'pota'`, TypeScript loads `exports.d.ts` which
references the same chain plus `public/components.d.ts`.

## How types reach src/ during development

The `files` array in `tsconfig.json` lists the ambient `.d.ts` files
that src/ JSDoc needs. These are loaded for type-checking but never
emitted to `generated/types/` (tsc skips `.d.ts` during emit).
Currently:

```json
"files": [
    "typescript/jsx/namespace.d.ts",
    "typescript/jsx/components.d.ts",
    "typescript/public/components.d.ts",
    "typescript/public/pota.d.ts"
]
```

## How to reference private types from src/

Private types use `export` and are imported via the `#type/*` alias
(defined in `package.json` `"imports"`):

```js
/** @type {import('#type/action.d.ts').action} */
```

This resolves to `typescript/private/action.d.ts`. The private types
are NOT in the `files` array — they're resolved as modules through the
import, not as ambient globals.

## public/ vs private/

- **public/** — ambient types, globally available. No `export`
  keyword. Consumers see these without importing.
- **private/** — module types with `export`. Only accessible via
  `import('#type/...')` in source JSDoc. Consumers never see these.

## Adding a new type

**Ambient (globally visible):** Add to `public/`, no `export` keyword.
Add to the `files` array in `tsconfig.json`.

**Private (src/ only):** Add to `private/`, use `export`. Reference
via `import('#type/yourfile.d.ts').YourType` in JSDoc. No tsconfig
changes needed.

**JSX-related:** Add to `jsx/`. If ambient, add to `files`.

## What NOT to put here

- Types that JSDoc/tsc can express — use `@type`, `@param`,
  `@template`, `@overload` etc. directly in the source.
- Generated output — that goes to `generated/types/` via tsc.
- Duplicates of types already in `generated/types/`.
